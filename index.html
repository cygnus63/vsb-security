Introduction

This project explains how modern hash functions become vulnerable and examines the types of attacks that can occur in real-world scenarios. It covers collision issues in MD5 and SHA-1, length extension weaknesses in Merkle–Damgård–based hashes, and the susceptibility of fast hash functions to brute-force attacks on GPU-accelerated systems. The project also demonstrates practical password-recovery workflows using Hashcat, including dictionary attacks, rule-based attacks, and comparisons with slow hashing algorithms such as bcrypt. Through these examples, the project highlights how secure algorithm choices and proper usage practices (such as salting and iteration) directly influence real-world security strength.

Current state of the art

MD5

Fast but broken for collision resistance. Deprecated for signatures.

SHA-1

Collision attacks demonstrated (SHAttered/Chosen-prefix). Avoid for integrity guarantees.

SHA-2 / SHA-256

Currently considered secure for collisions/preimage with correct usage.

SHA-3

Newer design (Keccak) with different structure; conservative choice.

PBKDF2 / bcrypt / scrypt / Argon2

Password hashing families that intentionally slow hashing (CPU / memory hardness).

Attack types

1. Collision Attacks

Definition

A collision occurs when two different messages, m1 and m2, produce the same hash value despite being distinct:

m1 ≠ m2, but H(m1) = H(m2).

Security Impact

This undermines integrity and digital signatures, because a signature typically applies to the hash rather than the full message. If an attacker can craft two different documents with identical hashes, they may replace a legitimate document with a malicious one without invalidating the signature.

Example

  • In 2017, researchers generated a collision on SHA-1 and produced two visually different PDF files that shared the same SHA-1 hash.
  • A signed "safe" PDF could be replaced with a "dangerous" one while preserving the signature's validity.

2. Preimage Attacks

First Preimage Attack

Given a hash value h, the goal is to find a message m such that H(m) = h.

If successful, this breaks the assumption that a hash uniquely commits to a specific message.

Second Preimage Attack

Given a specific message m₁, the attacker seeks a different message m₂ with the same hash:

m₁ ≠ m₂, but H(m₁) = H(m₂).

Security Impact

Preimage resistance is essential for commitment schemes, digital signatures, software distribution, and forensic validation.

Example

  • For weak hashes like MD5, large-scale GPU attacks can find second preimages significantly faster than brute force would suggest, making MD5 unsuitable for integrity-sensitive applications.

3. Length Extension Attacks

Definition

Hash functions based on the Merkle–Damgård construction (e.g., MD5, SHA-1, SHA-256) allow an attacker to compute:

H(message || padding || extra_data)

even when they do not know the original message, as long as they know its hash.

Security Impact

This breaks naïve authentication mechanisms such as:

H(secret_key || message)

because the attacker can append data and compute a valid extended hash without knowing the secret key.

Example

  • If an API validates requests with token = H(secret || "action=read"), an attacker can create a valid token for "action=read||delete" through length extension, enabling unauthorized operations.

4. Brute-Force and Dictionary Attacks

Definition

An attacker tests large numbers of candidate inputs until the correct one is found. Fast hashes (e.g., MD5, SHA-1, SHA-256) allow extremely high throughput on modern GPUs, making password recovery feasible if users choose weak passwords.

Security Impact

Unsalted or fast password hashes enable attackers to attempt billions of guesses per second.

This is a primary reason modern password storage relies on slow, memory-hard algorithms like bcrypt, scrypt, or Argon2.

Example

  • Using a GPU-based tool such as Hashcat, attackers can test over 50 billion MD5 hashes per second, making 8-character passwords trivial to recover.

5. Chosen-Prefix Collision Attacks

Definition

A more powerful variant of collision attacks where the attacker can choose two different prefix messages and generate controlled suffixes so that both combined messages produce the same hash value.

H(prefixA || suffixA) = H(prefixB || suffixB)

Security Impact

This enables forgery of high-value documents or certificates, because the attacker can insert arbitrary content at the beginning while still inducing a hash collision.

Example

  • Chosen-prefix collision attacks on SHA-1 (Marc Stevens, 2020) allowed the creation of two different PGP identity certificates sharing the same SHA-1 hash. This made it possible to "clone" trust from a legitimate certificate to a malicious one.

Hashcat

What Is Hashcat?

Hashcat is an open-source password recovery tool designed to crack hashed passwords using high-performance GPU-accelerated computation. It supports a broad range of hash algorithms, from simple formats such as MD5 to more complex schemes used in operating systems, encrypted containers, and authentication frameworks. Hashcat provides several attack modes, including dictionary, rule-based, mask, hybrid, and brute-force attacks. Its performance advantages stem from optimized parallel processing on modern GPUs, allowing rapid evaluation of password candidates. Hashcat is commonly employed in security auditing and digital forensics, and its use must always remain within legal and authorized environments.

Purpose of Using Hashcat

Hashcat is utilized in cybersecurity environments for several critical purposes:

Security Auditing

Organizations use Hashcat to evaluate the strength of their password policies. By attempting to crack password hashes, security teams can identify weak or predictable credentials and reinforce authentication requirements before an actual breach occurs.

Incident Response and Digital Forensics

During security incident investigations, analysts may need to recover passwords from compromised systems to determine the extent of unauthorized access. Hashcat enables the controlled recovery of password material from hash values, supporting evidence analysis and timeline reconstruction.

Password Strength Validation

When deploying new systems or applications, developers and security engineers use Hashcat to test whether implemented password rules are resilient against real-world attack methods. This helps verify that users are adequately protected from common credential-based attacks.

Security Research and Training

Hashcat provides a practical platform for studying hashing algorithms, attack techniques, and GPU-accelerated computation. It is widely used in academic and professional training environments to simulate offensive and defensive security scenarios.

Authorized Password Recovery

In legally approved contexts, Hashcat may be used to recover passwords that have been lost or forgotten when only the hash is available. This is strictly limited to systems for which explicit authorization has been granted.

How to use Hashcat?

1) Basic dictionary attack (MD5)

Command (example):

hashcat -m 0 -a 0 hashes_md5.txt rockyou.txt --status --force

Notes: -m 0 selects MD5. Fast on GPUs — demonstrates why raw MD5 is unsuitable for password storage.

2) Salted SHA-256 (dictionary + rule)

hashcat -m 1410 -a 0 salted_hashes.txt wordlist.txt -r rules/best64.rule --username

Use --username when the file format is user:hash:salt or similar. Iteration counts increase cost.

3) Slow password hash (bcrypt)

hashcat -m 3200 -a 0 bcrypt_hashes.txt wordlist.txt

bcrypt intentionally limits throughput; expect very low attempts/sec. Useful to compare performance numbers and demonstrate defensive value.

Conclusion

A recent incident occurred at SKT, one of the three major telecom companies in Korea, involving serious issues with password security. In this case, the problem wasn’t that the server passwords were protected by a weak hashing algorithm, but that they were stored without any encryption at all. Through this investigation and practice, I realized once again how alarming the security situation is, especially considering that a telecom provider used by more than half of the Korean population kept server passwords in plain text. As I will likely operate servers or websites in my future career, this incident reinforced my commitment to avoiding weak algorithms and instead using secure ones such as Argon2 or bcrypt to ensure proper protection.
